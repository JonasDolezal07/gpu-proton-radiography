ENERGY CONSERVATION AUDIT — prad Boris integrator
============================================================
Date:     2026-05-25
Hardware: Apple M4, MoltenVK Vulkan 1.3.323, prad v0.3.1
Test:     validate.py test 4 (uniform Bz = 1 T, 50,000 protons, 14.7 MeV)
          validate.py test 12 (zero field, 10,000 protons, 60 MeV)


FINDING: THE σ/μ ≈ 10⁻¹² WAS A HOST-SIDE ARITHMETIC ARTIFACT
--------------------------------------------------------------
The original test harness reported σ/μ ≈ 9.4×10⁻¹³ for test 4 and
1.4×10⁻¹⁴ for test 12.  Investigation showed:

  — All 50,000 particles in test 4 produce EXACTLY ONE distinct KE value
    in the output CSV (string "14.700003").  True physical σ = 0.

  — The nonzero σ/μ was produced entirely by the Python test harness:
    _mean() used Python's built-in sum(), which accumulates O(N × ε_f64)
    error when summing 50,000 identical float64 values.  The computed mean
    was offset from the true value by ~1.4×10⁻¹¹ MeV, and this offset
    propagated directly into _std() as a spurious nonzero result.

  Proof:
    Python sum() mean:        1.470000299998613e+01
    True value (all equal):   1.470000300000000e+01
    Offset:                   1.387e-11
    Python _std output:       1.387e-11           (= the offset, exactly)
    σ/μ (Python):             9.44e-13

    np.std of same array:     1.78e-15            (float64 numerical noise)
    σ/μ (numpy):              1.21e-16 ≈ ε_f64

  Fix: replaced _mean / _std with np.mean / np.std in the two energy
  tests, and added an n_unique_ke_values counter to the report.

After fix:
  Test 4:   std/mean = 1.21e-16,  unique KE values = 1  (PASS)
  Test 12:  std/mean = <1e-14,    unique KE values = 1  (PASS)


PHYSICAL EXPLANATION: WHY ALL VALUES ARE IDENTICAL
---------------------------------------------------
In a spatially uniform B field, the Boris push for every particle performs
exactly the same sequence of floating-point operations at every step:

  1. Field sample: B = (0, 0, 1 T) everywhere — same for all particles.
  2. Boris rotation: t = (q/m)(B)(dt/2)/γ — depends only on |u| (same for
     all monoenergetic particles) and B (same everywhere).
  3. The rotation preserves |u|² exactly in an E=0 field (the Boris theorem).
  4. Position update: x += (u/γ) dt — depends on u and position, but
     position does not feed back into the uniform field value.

Steps 1–3 are position-independent in a uniform field.  Since all particles
start with the same |u| and experience the same B at every step, they perform
identical float32 arithmetic regardless of trajectory.  The GPU scheduler
assigns different work groups to different particles, but the floating-point
operations within each work group are the same — so all particles produce
the exact same float32 |u|² at the detector (regardless of when they hit or
how many steps they took).

The KE formula in the shader:
    gamma = sqrt(1.0 + dot(vel, vel) * INV_C_SQ)
    energy = (gamma - 1.0) * C_M_S * C_M_S
evaluates to the same float32 for all particles.

This is expected and confirmed.  It is NOT a coincidence or a sign that the
computation is wrong — it is the direct consequence of the Boris algorithm
being position-independent in a uniform field.


CHECKLIST ITEMS
---------------

A. Host-side float64 accumulation
   STATUS: FOUND AND FIXED.
   Python _mean() via sum() introduced O(N × ε) error in the mean,
   producing spurious nonzero std for identical values.
   Fix: use np.mean / np.std.  New σ/μ = 1.21e-16 ≈ ε_f64.

B. Array reuse / shared-memory bug
   STATUS: NOT AN ISSUE.
   All particles produce the same result independently; no array aliasing
   detected.  The GPU SSBO (Shader Storage Buffer Object) for particles
   uses per-invocation state stored in registers throughout all steps.

C. Cross-platform consistency
   STATUS: PARTIALLY ASSESSED.
   Test 4 on Apple M4 (MoltenVK 1.3.323): unique KE = 1, σ/μ = 1.21e-16.
   RTX 4090 / Linux (from benchmarks/validation/nvidia_rtx4090_ubuntu2204.txt):
   not separately assessed for energy conservation — this is a TODO for
   the NVIDIA validation log.  However, the result (all identical) follows
   from the field-independence argument above and should hold on any
   IEEE 754-compliant implementation.

D. Shader precision — MoltenVK/Metal op promotion
   STATUS: NOT DETECTED.
   The shader uses float (float32) throughout.  The suspicious σ/μ ≈ 10⁻¹²
   was explained entirely by the Python host-side artifact (item A), not by
   any GPU precision promotion.  If Metal were silently promoting the Boris
   arithmetic to float64, we would see σ/μ < ε_f32 but > 0 across particles
   (particles at different positions would have slightly different KE due to
   slightly different trajectories).  Instead we see σ = 0 exactly, which is
   consistent with pure float32 (all particles identical) rather than float64
   promotion (which would show inter-particle spread).
